## What was built

Market Intelligence's recommendations are no longer hand-authored mock text — they're computed
by a new, deterministic `StrategyEngine` that runs three independent strategies against every
tracked instrument. **No AI, no live data, no network calls, no broker integration, no autonomous
trading.** Every recommendation is reproducible: the same instrument snapshot always produces the
same strategy results, the same overall call, and the same explanation.

- **Three independent strategies**, each a pure function of a shared `StrategyContext`:
  - **Moving Average Crossover** — BUY when a short-term average is above a long-term average and
    price confirms it; SELL for the mirror case; HOLD otherwise.
  - **RSI Reversal** — SELL when RSI is overbought (≥70), BUY when oversold (≤30), HOLD in
    between. Deliberately a *reversal* read, so it can disagree with trend-following strategies on
    the same instrument — and often does.
  - **Momentum** — BUY/SELL when price change, volume, and trend all confirm the same direction;
    HOLD if they don't.
- **`StrategyEngine`** runs all three against every instrument and aggregates them into a
  `StrategyScore`: an overall signal, an overall recommendation (mapped onto the existing 5-level
  `Recommendation` type from Build 0.3.0), an overall confidence, and an **agreement level** —
  Strong Agreement (unanimous) → Moderate Agreement (majority + a neutral dissenter) → Mixed
  Signals (majority + an opposite-direction dissenter) → Conflict (no majority at all, a genuine
  three-way split).
- **Market Intelligence** gained three new sections per opportunity: **Generated by** (each
  strategy's signal and confidence, plus the overall call), **Strategy breakdown** (full evidence
  and each strategy's percentage contribution to the combined confidence), and **Agreement** (the
  level plus a plain-English explanation of why). The existing Recommendation panel, Decision
  Breakdown, and Intelligence Score sections are unchanged in structure — only the
  recommendation/confidence/narrative/evidence feeding the Recommendation panel are now
  engine-derived instead of static mock text.
- **Watchlist** and the **Dashboard's watchlist snapshot** gained a "Primary strategy" column —
  the highest-confidence individual strategy for that instrument, regardless of whether it agrees
  with the overall call (worth surfacing either way — see TSLA below).
- **Dashboard** gained a Strategy Summary card: strategies evaluated (15 = 3 strategies × 5
  instruments), average confidence, agreement distribution, and the single highest-confidence
  strategy result across every instrument.
- **Trade Journal** now permanently records, for Market-Intelligence-sourced trades: the primary
  strategy, the agreement level, the overall confidence, and a one-line evidence summary — visible
  in the trade's Market Intelligence context block alongside its existing recommendation/evidence.
- **System Health** gained a live Strategy Engine panel (Running, Strategies loaded, Evaluation
  time) — and the old static "Strategy Engine: Running" row in the Services list was removed,
  since it's now genuinely redundant with the live panel (same de-duplication pattern used for
  Market Data and Persistence in Builds 1.0.0/0.9.0).

## Architecture overview

```
StrategyContext          — deterministic per-instrument inputs (short/long MA, RSI, volume
                            ratio, trend), all derived from the instrument's existing mock
                            snapshot; no historical price series exists in this prototype
Strategy (interface)     — { id, name, evaluate(context): StrategyResult }
  ├── MovingAverageCrossoverStrategy
  ├── RsiReversalStrategy
  └── MomentumStrategy
StrategyResult            — one strategy's verdict: signal, confidence, evidence[]
StrategyEngine            — runs every registered strategy against an instrument, aggregates
                            the results into a StrategyScore (majority vote → overall signal;
                            confidence-weighted → overall confidence; vote pattern → agreement
                            level; highest individual confidence → primary strategy)
StrategyScore             — the engine's per-instrument aggregate output
```

No resilient-fallback wrapper, no async hook, no configuration — unlike `MarketDataProvider` or
the persistence layer, the engine is a pure, synchronous, in-memory computation with nothing to
fail against. `getStrategyEngine()` is a plain module singleton; server components call
`evaluateAll(instruments)` directly and pass the result down as props, the same way market data
quotes are threaded through — no new client-side state was needed anywhere except where a page was
already a client component for other reasons.

**Why `Strategy` isn't exported from the shared `@/lib/types` barrel:** that barrel already
exports an unrelated `Strategy` type — the Strategies page's mock rule metadata (Momentum
Breakout, Mean Reversion, Trend Following, Volatility Filter), untouched since Build 0.1.0 and
completely out of scope for this build. Same word, two different concepts from two different
builds. Rather than overload one name, the engine's `Strategy` interface lives in
`src/lib/strategy-engine/strategy.ts`, imported directly by the engine's own files; everything
else it needs (`StrategyContext`, `StrategyResult`, `StrategyScore`, `AgreementLevel`,
`StrategySignal`) lives in `@/lib/types/strategy-engine.ts` since none of those names collide.

**Why the Strategies/Signals pages are untouched:** they're a separate, pre-existing mock system
(Build 0.1.0/0.2.0) unrelated to Market Intelligence's opportunities. This build's engine only
powers Market Intelligence, Watchlist's Primary Strategy column, the Dashboard's Strategy Summary,
and Trade Journal metadata for Market-Intelligence-sourced trades — Signal-sourced trades
(placed from the Signals page) still use the older mock strategy name and never populate the new
Trade Journal fields.

**On the deterministic indicator proxies:** this prototype has no historical price series for any
instrument — only a single current snapshot (price, day change, volume). Short/long moving
averages are proxied from price minus a multiple of today's absolute change; RSI is proxied by
mapping today's percent change onto the 0-100 scale around a neutral 50; volume ratio compares
today's volume against a fixed baseline, independent of price change so Momentum's volume
confirmation isn't just a restatement of its own price input. None of this is random — the same
instrument snapshot always produces the same context, the same strategy results, and the same
overall call.

**A deliberate, disclosed consequence:** since recommendations are now computed rather than
hand-authored, some instruments' calls changed from their old static values. TSLA's mock data (a
strong daily gain, elevated volume) now reads as bullish to Moving Average Crossover and Momentum,
against a bearish RSI Reversal dissent — "Buy" with Mixed Signals agreement, rather than the old
static "Strong Sell." This is the intended effect of replacing hand-authored recommendations with
a real, if simple, engine — not a bug to fix by tuning constants to match the old narrative.

## Database changes

**`0009_strategy_engine.sql`** — adds `primary_strategy` (text), `strategy_agreement` (text,
checked against the four agreement levels), `overall_confidence` (numeric, 0-100), and
`evidence_summary` (text) to `paper_trades`. All four nullable and purely informational — never
read by any P/L calculation — so Signal-sourced trades and every trade placed before this
migration are unaffected; they simply have all four as null.

## Manual Supabase steps required

Run `0009_strategy_engine.sql` in the SQL Editor (same anon-key limitation as every schema change
in this project). Without it, a signed-in Market-Intelligence trade insert fails on the unknown
columns, which `ResilientPaperTradeStore` treats as a generic failure and falls back to local
storage for the rest of that session — not data loss, but not the intended behaviour.

**Also worth noting:** while testing this build, an unauthenticated insert attempt against the
live connected project was genuinely rejected with a row-level-security violation, confirming
Build 1.1.0's `0006`/`0007` migrations are still active (as already established during Build
1.2.0). `0009` is additive on top of that.

## Strategy Engine design notes

- **Aggregation assumes exactly 3 strategies.** A majority of 2 always means exactly one
  dissenter, and a leading count of 1 always means all three signals differ — both are relied on
  directly in `aggregateResults`. Registering a fourth strategy would need this re-derived, not
  just reused as-is (documented inline where the assumption is made).
- **`Recommendation`'s existing sell asymmetry is preserved, not changed.** The type from Build
  0.3.0 has two buy tiers (Strong Buy, Buy) but only one sell tier (Strong Sell) — a pre-existing
  design choice. Every overall SELL signal maps to "Strong Sell" regardless of confidence or
  agreement; this build didn't add a second sell tier to avoid touching that established type and
  its existing consumers (`isTradeableRecommendation`, `sideForRecommendation`).
- **"Primary strategy" is the highest-confidence individual result, even if it dissents.** For
  NVDA, Momentum (89%, agrees) is primary; for TSLA, RSI Reversal (the dissenting SELL call) is
  primary because its confidence is highest. Surfacing the dissenter as "primary" when it has the
  strongest individual conviction is intentional — it's the most analytically interesting
  strategy for that instrument, not necessarily the one that won the vote.
- **Evaluation time is measured, not estimated**, and shown honestly — typically under 1ms for 5
  instruments × 3 strategies, since it's pure JS math with no I/O. `performance.now()` is called
  inside `StrategyEngine.evaluateAllWithTiming()`, not inside the System Health page component
  itself — React's purity rule (no impure calls directly in a component body) otherwise flags a
  direct call, correctly, since the page component must stay idempotent.

## What is intentionally not included yet

- No AI, no machine learning, no live/real market data feeding the strategies — same three
  deterministic rules every time, exactly as instructed.
- No broker integration, no live order placement, no autonomous trading, no "Hermes" — exactly as
  instructed.
- Signal-sourced trades (from the Signals page) don't run through the engine and never populate
  the new Trade Journal fields — a separate, older mock system, out of scope for this build.
- No historical price series or charting — every indicator is a proxy derived from a single
  current snapshot, not a real moving average over real historical closes.
- No per-user or per-account strategy configuration — the same three strategies with the same
  thresholds run for everyone.
- Exit/close price provenance is still not recorded (only entry price provenance, from Build
  1.2.0) — a future build could extend `useCloseTradeFlow` similarly.

## How to run the prototype

```bash
cd Trading/platform/web
npm install
npm run dev
```

No environment variables are required. `npm run lint` and `npm run build` both pass cleanly.

**Verified in this build (local prototype mode):** Market Intelligence's Generated By, Strategy
Breakdown, and Agreement sections render correctly and match hand-calculated values exactly (e.g.
NVDA: Moving Average Crossover BUY 74%, RSI Reversal SELL 78%, Momentum BUY 89%, overall Buy at
82% confidence, Mixed Signals); placing a trade from Market Intelligence persists all four new
fields correctly into `localStorage`; Trade Journal displays them; closing the trade still works;
Watchlist and the Dashboard's watchlist snapshot show the correct Primary Strategy per instrument
(matching hand-calculated values, including TSLA and NVDA correctly showing their dissenting
highest-confidence strategy as primary); the Dashboard's Strategy Summary card shows correct
totals (15 strategies evaluated, agreement distribution 0/3/2/0 matching the 5 mock instruments'
actual outcomes); System Health's new Strategy Engine panel shows a real, sub-millisecond
evaluation time; the old Strategies/Signals pages are completely unaffected; gating with Supabase
configured continues to work correctly with no regressions from this build's changes.

**Not verified against a real authenticated Supabase session:** migration `0009` has not been
applied to the live project, and (as disclosed in Builds 1.1.0/1.2.0) no confirmable test account
was available in this environment — so a Market-Intelligence trade actually persisting its
strategy metadata to Supabase was not exercised end-to-end.
