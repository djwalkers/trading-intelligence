# Trading Intelligence â€” Web Prototype

Build 1.12.0. A dark-themed platform for monitoring signals and paper trading performance, built with Next.js
(App Router), TypeScript, and Tailwind CSS. The platform's philosophy: **understand first, decide
second, trade last** â€” every recommendation explains its reasoning and what would change it.
Signal and strategy data is mocked â€” there is no broker connection and no live trading. Paper
trades are persisted through a storage-agnostic abstraction: `localStorage` by default, or Supabase
when configured (see [Persistence mode](#persistence-mode) below). Instrument prices flow through a
similar storage-agnostic abstraction: mock data by default, or a live external market data provider
when configured (see [Market data mode](#market-data-mode) below). When Supabase is configured, the
whole app additionally requires sign-in, and paper trades are scoped to the signed-in user (see
[Authentication](#authentication) below) â€” in local prototype mode (no env vars), none of this
applies and the app behaves exactly as before. "Trading Intelligence" is a temporary product name
for this prototype phase.

## Authentication

When Supabase is configured (see [Persistence mode](#persistence-mode) below), the whole app is
gated behind email/password sign-in via a new `AuthGate` (`src/components/layout/AuthGate.tsx`),
backed by `AuthProvider`/`useAuth()` (`src/lib/auth/auth-context.tsx`). In local prototype mode
(no env vars), there is no auth concept at all â€” every page renders unconditionally, exactly as
in every prior build.

**Sign up / sign in** at `/sign-up` / `/sign-in` â€” plain email/password via Supabase Auth, no
OAuth or magic links. **Sign out** is in the sidebar, next to the signed-in user's email.
**Session expiry**: if a signed-in session lapses (rather than the user choosing to sign out), the
sign-in page shows "Your session has expired. Please sign in again."

**Forgot your password?** `/forgot-password` sends a reset email (a generic "check your inbox"
message either way, so as not to reveal which addresses have accounts); the emailed link lands on
`/reset-password`, which relies on Supabase establishing a temporary recovery session from the URL
â€” if there isn't one (invalid, already used, or expired link), it says so instead of showing a
broken form.

Paper trades are scoped to the signed-in user, both by explicit filtering in
`SupabasePaperTradeStore` and by Row Level Security server-side (see
[`docs/database/SUPABASE-SETUP.md`](../../docs/database/SUPABASE-SETUP.md) for the migrations
this requires). If a write is attempted with no session, `SupabasePaperTradeStore` throws a
distinct `AuthRequiredError` that `ResilientPaperTradeStore` recognises and does **not** fall back
to local storage for â€” silently saving to an unscoped store would be wrong for a user-scoped app.

System Health's Authentication panel always reflects the real, current state: Auth
(Enabled/Disabled), Current user (email, when signed in), and Data scope (User scoped / Local
prototype).

## Persistence mode

Paper trades are saved through a small storage-agnostic interface (`PaperTradeStore`,
`src/lib/persistence/paper-trade-store.ts`) with two real implementations â€”
`LocalStoragePaperTradeStore` and `SupabasePaperTradeStore` â€” chosen by `getPaperTradeStore()`:

```bash
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
```

**Both unset (default): local storage, no configuration needed, no sign-in.** `npm run dev` just
works.

**Both set:** the app reads and writes paper trades to Supabase, scoped to the signed-in user (see
[Authentication](#authentication) above) â€” copy `.env.example` to `.env.local`, fill these in from
your project's API settings, and see
[`docs/database/SUPABASE-SETUP.md`](../../docs/database/SUPABASE-SETUP.md) for the full walk-through
(create a project, run all sixteen migrations, create an account, verify the tables). Only the
public anon key is ever used, client-side â€” never a service role key; access control is delegated
to the RLS policies already in the migrations. As of Mission 6, an optional
`SUPABASE_SERVICE_ROLE_KEY` (deliberately not `NEXT_PUBLIC_`-prefixed) exists for a future
background worker only â€” nothing in the browser app reads it, and nothing in the running app calls
it yet; see [Server architecture preparation](#server-architecture-preparation-mission-6) below.

**If Supabase is configured but unreachable**, the app does not break: `ResilientPaperTradeStore`
catches the failure, logs it to the console, falls back to `localStorage` for the rest of the
session (no repeated retries), and shows a small banner â€” "Persistence unavailable. Falling back
to local storage." â€” until the next page load. System Health's Persistence panel always reflects
the real, current state: mode, connection, and last synchronisation time.

If this browser already has local paper trades and the signed-in user has none in Supabase yet, a
one-time modal offers to import that history â€” answering either way (Import or Skip) means it's
never asked again on that browser.

The schema itself â€” real, runnable SQL, not just documentation â€” lives in `supabase/migrations/`
(sixteen files, numbered in run order â€” the first five from Build 0.7.0, `user_id` and user-scoped
RLS from Build 1.1.0, entry price provenance columns from Build 1.2.0, Strategy Engine metadata
columns from Build 1.3.0, Bot Runner metadata columns from Mission 1, the bot scan id column from
Mission 1.1, Portfolio Risk Manager metadata columns from Mission 2, Position Manager metadata
columns from Mission 3, Mission 6's two dormant worker-only tables (`bot_schedules` and
`bot_decisions`, not read or written by the browser app), and â€” Mission 7 â€” `decision_history`, a
new table read and written by the browser today, unlike Mission 6's two) and `supabase/seed.sql`
(sample data for manually poking at the schema; not read by the app). See
[`docs/database/SUPABASE-PERSISTENCE-PLAN.md`](../../docs/database/SUPABASE-PERSISTENCE-PLAN.md)
for the schema rationale, and
[`infrastructure/supabase/README.md`](../../infrastructure/supabase/README.md) for the
infrastructure-level overview.

## Market data mode

Instrument prices are served through a small storage-agnostic interface (`MarketDataProvider`,
`src/lib/market-data/market-data-provider.ts`) with two real implementations â€”
`MockMarketDataProvider` and `ExternalMarketDataProvider` â€” chosen by `getMarketDataProvider()`:

```bash
NEXT_PUBLIC_MARKET_DATA_PROVIDER=
NEXT_PUBLIC_MARKET_DATA_API_KEY=
```

**Both unset (default): mock prices, no configuration needed.** `npm run dev` just works, and
Watchlist/Portfolio/Settings/Operations Centre all show a "Mock" source and "Mocked" mode.

**Both set:** the app fetches live quotes from Finnhub's quote endpoint â€” copy `.env.example` to
`.env.local` and fill these in with a free [finnhub.io](https://finnhub.io) API key.
`NEXT_PUBLIC_MARKET_DATA_PROVIDER` is currently a display label only (e.g. `Finnhub`), not a
multi-vendor selector â€” see [`ExternalMarketDataProvider`](src/lib/market-data/external-market-data-provider.ts)
to add another vendor.

**If the external provider is configured but fails**, the app does not break:
`ResilientMarketDataProvider` catches the failure, logs it to the console, falls back to mock
prices for the rest of the session (no repeated retries), and the Operations Centre's Market Data
group shows mode "Fallback" with the failure reason. The Dashboard's Market Overview section always
shows a condensed, real-time read of the current provider and whether fallback is active; the full
detail (provider, mode, last updated, instruments loaded) lives on the Operations Centre.

As of Build 1.2.0, trade **entry** prices (placed from Signals or Market Intelligence) also go
through this same provider â€” placing a trade fetches a live quote at that moment, and the
confirmation modal shows the price's source (Mock/External), provider, and last-updated time, plus
a note if the provider had fallen back to mock for that quote. This provenance is stored on the
resulting trade (`entryPriceSource`, `entryPriceProvider`, `entryPriceTimestamp`, all optional â€”
trades from before this build have none of them and load exactly as before).

## Strategy Engine

Market Intelligence's recommendations are computed, not hand-authored. `StrategyEngine`
(`src/lib/strategy-engine/`) runs three independent, deterministic strategies â€” **Moving Average
Crossover**, **RSI Reversal**, and **Momentum** â€” against every tracked instrument, then aggregates
them into an overall signal, confidence, recommendation, and **agreement level** (Strong Agreement
â†’ Moderate Agreement â†’ Mixed Signals â†’ Conflict, based on how much the three strategies agree).

The synchronous path (`evaluateAll`/`evaluateInstrument`) has no configuration, no network calls,
no failure mode â€” every strategy is a pure function over already-in-memory mock instrument data,
with moving averages, RSI, and volume ratio computed as deterministic proxies derived from each
instrument's current snapshot (`buildStrategyContext`, `build-context.ts`). The same instrument
always produces the same result. This is what Market Intelligence's Generated By / Strategy
Breakdown / Agreement sections, Watchlist's Primary Strategy column, and the Dashboard's Strategy
Summary card all still read from.

As of Mission 9, a second, async path exists â€” `evaluateAllWithHistory`/
`evaluateInstrumentWithHistory` â€” that computes the same fields from **real** 90-day OHLCV history
(via `buildStrategyContextFromHistory`) whenever enough is available, falling back to the snapshot
proxies per-instrument otherwise. The Bot Runner (both the browser and the Mission 8 VPS worker)
uses this path; the display pages above don't yet â€” see [Historical market
data](#historical-market-data-mission-9) below for the full design and why that scoping line was
drawn there.

For trades placed from Market Intelligence, permanent metadata is still recorded on the resulting
`PaperTrade` (`primaryStrategy`, `strategyAgreement`, `overallConfidence`, `evidenceSummary`, all
optional). The Signals/Strategies pages are a separate, older mock system (Build 0.1.0) and don't
use this engine at all.

## Historical market data (Mission 9)

A second data layer, parallel to [Market data mode](#market-data-mode) above â€” instead of live
current-price quotes, this one serves 90 days of daily OHLCV candles per instrument, so the
Strategy Engine can compute *real* SMA/EMA/RSI/momentum/volume-ratio values instead of proxying
them off a single day's snapshot. Same architecture as market data mode, file-for-file:
`HistoricalMarketDataProvider` interface, `MockHistoricalMarketDataProvider` (default), and
`ResilientHistoricalMarketDataProvider` (external when configured, mock fallback if it ever fails,
tracked status).

**Real candles now come from Alpha Vantage, server-only (Maintenance 1.11.2)** â€” see
[Real historical data](#real-historical-data-maintenance-1112) below. Finnhub's historical candle
endpoint (`ExternalHistoricalMarketDataProvider`) has been removed; Finnhub is still used for live
current-price quotes only (see [Market data mode](#market-data-mode)).

**Mock candles are deterministic, not random**: a seeded PRNG (seeded from each symbol's own name)
generates a 90-day random walk, then the whole series is rescaled so its final close lands exactly
on that instrument's current mock snapshot price â€” same instrument in, same 90-day history out,
every run, every build, matching this app's existing "no randomness anywhere" convention for mock
data.

Six new pure, reusable indicator functions live in `src/lib/indicators/`: `calculateSMA`,
`calculateEMA`, `calculateRSI` (Wilder's construction), `calculateMomentumPercent`,
`calculateVolumeRatio`, `calculateVolatility` â€” each takes a plain number series and a period,
returns the latest value or `null` if there isn't enough data yet.

`buildStrategyContextFromHistory()` (`src/lib/strategy-engine/build-context.ts`) uses these to
compute the same `StrategyContext` fields the original snapshot-proxy `buildStrategyContext()`
always produced â€” EMA(12)/SMA(30) for the moving-average pair, RSI(14), a 20-day volume ratio, and
(new) a 5-day `momentumPercent` field the Momentum strategy now reads instead of
`instrument.changePercent` directly. Falls back to the snapshot proxy per-instrument when there
isn't enough history (fewer than 31 candles) â€” never a hard failure. **Zero changes** were needed
to the Moving Average Crossover or RSI Reversal strategies themselves; they already read these
fields from context and don't know or care whether the values are real or proxied.

Wired into the Bot Runner only (`runBotScan()` now calls the new async
`StrategyEngine.evaluateAllWithHistory()` instead of the synchronous `evaluateAll()`) â€” both the
browser and the Mission 8 VPS worker share this one call site, so neither duplicates the upgrade.
The Dashboard/Market Intelligence/Watchlist display pages deliberately still use the synchronous,
snapshot-proxy path â€” a disclosed scoping decision, not an oversight; see
[`../../docs/product/MISSION-9-HISTORICAL-MARKET-DATA.md`](../../docs/product/MISSION-9-HISTORICAL-MARKET-DATA.md)
for why. System Health gained a matching "Historical Data" status panel (provider, connection mode,
instruments loaded, last refresh, cache age) alongside the existing Market Data one.

## Real historical data (Maintenance 1.11.2)

`AlphaVantageHistoricalMarketDataProvider` (`src/lib/market-data/`) calls Alpha Vantage's
`TIME_SERIES_DAILY` endpoint (`outputsize=compact`, ~100 trading days â€” comfortably covers the
90-day lookback above) for the app's five-symbol universe (AAPL, MSFT, NVDA, TSLA, SPY), parsing
`"Time Series (Daily)"` into the existing `OHLCVCandle` model and detecting five distinct failure
modes (invalid key, rate limit, missing symbol, malformed response, HTTP failure) as a typed
`AlphaVantageError`.

**Server-only, by construction, not just by convention**: `ALPHA_VANTAGE_API_KEY` is never
prefixed `NEXT_PUBLIC_`, and the provider file itself is `import "server-only"`. It's constructed
by exactly one factory, `get-server-historical-market-data-provider.ts` (also server-only), which
is imported by exactly one caller, `src/worker/process-schedule.ts` â€” a file the Next.js browser
bundle never includes (the worker runs via `tsx`, entirely outside the Next.js build). The
browser's own factory, `get-historical-market-data-provider.ts`, was changed to always resolve to
Mock â€” there is no historical provider it can safely construct client-side anymore, since Finnhub's
historical endpoint is gone and Alpha Vantage's key must never reach the browser. **This means the
browser's own manual "Run Bot Scan" always uses mock candles now; only the VPS worker's scheduled
scans use real Alpha Vantage data.**

**Caching, not a request per scan**: each symbol's series is cached for 24 hours, backed by a JSON
file on disk (`.data/alpha-vantage-historical-cache.json`, gitignored) rather than only in memory,
so a worker restart doesn't immediately re-fetch every symbol â€” the mission's "retain cached data
across restarts where practical" requirement, met without any Supabase schema change. In steady
state this keeps the worker to at most one Alpha Vantage request per symbol per day, regardless of
how often a schedule scans (every 15/30/60 minutes) â€” verified live (see
[`../../docs/product/MAINTENANCE-1.11.2-REAL-MARKET-DATA.md`](../../docs/product/MAINTENANCE-1.11.2-REAL-MARKET-DATA.md)).

**Why the System Health "Historical Data" panel still shows Mock**: it reads the browser's own
factory, which â€” as above â€” never resolves to Alpha Vantage. This is the same disclosed
browser/worker visibility gap as the Server Scheduler panel (Mission 10): the browser has no live
channel into a separate worker process. The real Alpha Vantage status (source, last refresh,
symbols loaded, cache age, fallback reason) is reported instead as one plain-English
`historical_data_status` line in the worker's own log output after every scan.

## Bot Runner

The first autonomous paper-trading loop â€” manually triggered, or (as of Mission 4) run on a
browser-based schedule. Clicking "Run Bot Scan" on the Dashboard â€” or a running schedule's timer
firing â€” calls `runBotScan()` (`src/lib/bot/bot-runner.ts`), which runs every watchlist instrument
through the Strategy Engine, ranks the tradeable opportunities by confidence, and then walks down
that ranked list, evaluating three tiers for each candidate in turn:

1. **Individual risk** (four hardcoded rules: max one trade per scan, minimum 75% confidence, no
   trading on Conflict agreement, a hard ÂŁ250 max notional per trade).
2. **Position Manager** (Mission 3) â€” classifies the candidate against any existing position in
   that instrument as `NEW_POSITION`, `ADD_TO_POSITION` (confidence improved by 5+ points over the
   last Bot trade there, agreement not weaker, position value â‰¤ ÂŁ750, 30+ minutes since the last
   trade), `HOLD_POSITION` (soft â€” not enough new evidence yet), or `BLOCK_POSITION` (hard â€” an
   opposing existing position, the value cap exceeded, or portfolio risk failing).
3. **Portfolio risk** (Mission 2, six hardcoded rules: max 5 open trades, max 60% of starting
   capital deployed, max 30% exposure to one sector, max 3 open trades per sector, minimum ÂŁ1,000

portfolio-risk failure causes the bot to fall back to the next-ranked candidate, same as an
individual-risk failure (Mission 1.1). Only a candidate that clears all three tiers opens a real

its trigger type (Manual/Scheduled, Mission 4), a portfolio exposure snapshot at scan time, every

## Bot scheduling

The Bot Runner can run on a schedule instead of only manually â€” Manual only (the default), Every
15 / 30 / 60 minutes â€” controlled from the Dashboard's Bot Runner panel (mode selector, Start/Stop,
next/last scan time, status). **This is browser-based scheduling, not a background service**: the
schedule's state (`BotSchedulerProvider`) is shared and readable from any page, but the actual
timer only advances while the Dashboard's `BotRunnerPanel` is mounted â€” a 10-second poll checks
whether the configured interval has elapsed and, if so, runs a scan. Close the tab or navigate
away and the countdown simply stops progressing until you return; if the scheduled time has already
passed when you come back, the next poll runs the "overdue" scan immediately rather than skipping
it silently.

Every scheduled scan goes through the exact same individual/Position Manager/portfolio risk
pipeline as a manual one â€” `runBotScan()` doesn't know or care what triggered it â€” and the schedule
stops itself automatically if the user is signed out (Supabase configured, no session) or if
persistence has fallen back away from Supabase, rather than continuing to run unattended in either
state. True 24/7 scheduling â€” independent of any open browser tab â€” would need a VPS or background
worker, which this mission deliberately does not build; see
[`../../docs/product/MISSION-4-SCHEDULED-BOT-SCANS.md`](../../docs/product/MISSION-4-SCHEDULED-BOT-SCANS.md)
for exactly what that would require.

## Decision Intelligence (Mission 7)

The bot's long-term analytical memory, separate from (and complementary to) the operational logs
above. Every candidate a scan evaluates â€” whether it opens a paper trade or gets rejected â€”
produces one `DecisionRecord` (`src/lib/decision-intelligence/types.ts`): identity (scan id,
timestamp, trigger type, rank), the opportunity (symbol, sector, side, entry price), the strategy
behind it (which strategy led, agreement, confidence, evidence summary), the portfolio's state at
that moment (deployed capital, available cash, sector exposure, total open trades), the decision
itself (action taken, rejection reason, Position Manager action, portfolio risk result), and an
outcome â€” `Pending` until the linked trade closes, then automatically classified as `Win`/`Loss`/
`Neutral` (Mission 11, see [Outcome analysis](#outcome-analysis-mission-11) below). Rejected
candidates are never classified â€” this mission (7) records evidence for them, it doesn't judge
whether a decision was good or bad.

**Why rejected candidates matter as much as accepted ones**: a `PaperTrade` only ever exists for the
one candidate (if any) that actually opened a position in a scan. A future Hermes needs to learn
from the ideas that *didn't* clear risk checks too â€” what confidence/agreement combinations tend to
get rejected, and why â€” not only from trades that were placed. `buildDecisionRecords()`
(`src/lib/decision-intelligence/build-decision-records.ts`) derives one record per entry in a
completed scan's candidate list, so a scan that fails four candidates before the fifth opens a trade
produces five records, not one.

This is wired into the same shared `executeBotScan()` orchestration Mission 6 introduced â€” see
[Server architecture preparation](#server-architecture-preparation-mission-6) below â€”
via one more `BotExecutionContext` method, `persistDecisionRecords()`, called immediately after
every scan. Unlike Mission 6's `bot_schedules`/`bot_decisions` tables (deliberately dormant,
worker-only), the new `decision_history` table is live and used by the browser today: a new
`DecisionHistoryProvider` (mirrors `PaperTradesProvider`'s auth-identity re-hydration exactly) reads
and writes through a `DecisionHistoryStore` â€” Supabase when configured, local storage as the
fallback, the same resilient-fallback pattern every other persisted feature in this app uses.

The **Decision Intelligence** page (`/decision-intelligence`) shows every record in one simple,
filterable table â€” no charts â€” with independent filters for strategy, agreement, symbol, action
(Trade Opened / Rejected), and confidence band. System Health gained a matching status block
(status, records stored, last recorded). Every record carries a `version` field
(`DECISION_RECORD_SCHEMA_VERSION`), so a future Hermes build can evolve the shape without a breaking
rewrite of history already recorded. See
[`../../docs/product/MISSION-7-DECISION-INTELLIGENCE.md`](../../docs/product/MISSION-7-DECISION-INTELLIGENCE.md)
for the full design, including the `decision_history` table's `update` RLS policy â€” provisioned
here, first used by Mission 11's outcome analysis.

## Server architecture preparation (Mission 6)

Everything in this section is prepared, unused infrastructure â€” no VPS or background worker is
deployed, and nothing in the running app calls any of it yet. It exists so a future Mission 7
worker doesn't start from scratch. The browser Bot Runner and browser scheduling described above
are completely unchanged in behaviour.

`runBotScan()` itself was already server-safe (pure computation, no persistence, no browser API).
What was browser-bound was the orchestration around it â€” loading trades, saving the resulting
trade and decision â€” so Mission 6 extracted that into a shared `executeBotScan()` wrapper
(`src/lib/bot/bot-execution-context.ts`) taking a small `BotExecutionContext` (`loadTrades`/
`persistTrade`/`persistDecision`). The Dashboard's `BotRunnerPanel` now calls `executeBotScan()`
with a context backed by `usePaperTrades()`/`useBotDecisionLog()`, instead of calling `runBotScan()`
and persisting the result directly â€” behaviourally identical, verified manually (see below).

A parallel server-side implementation of that same context
(`src/lib/bot/server-execution-context.ts`) exists for a future worker, backed by:

- A service-role Supabase client (`src/lib/supabase/service-role-client.ts`) reading a new,
  optional `SUPABASE_SERVICE_ROLE_KEY` env var â€” deliberately not `NEXT_PUBLIC_`-prefixed, so
  Next.js never inlines it into the browser bundle.
- `src/lib/persistence/server-paper-trade-store.ts` â€” mirrors `SupabasePaperTradeStore`'s exact row
  mapping (the two now share `toDbTrade`/`fromDbTrade`/`PaperTradeRow`, exported for this purpose),
  but takes an explicit `userId` instead of deriving one from a browser session.
- `src/lib/scheduler/server-bot-decision-store.ts` â€” writes to the new `bot_decisions` table.

Every one of these server-only files starts with `import "server-only"` â€” the official Next.js
package that turns "don't import this from a client component" into a **build-time error**, not
just a convention, if a client component ever pulls one in (even transitively). For that reason
they are deliberately **not** re-exported from `src/lib/bot/index.ts` (the client-safe barrel) â€”
only imported directly by path from server-only call sites.

Two new, RLS-protected Supabase tables exist for a future worker to read/write, neither used by the
browser app: **`bot_schedules`** (one row per user â€” `enabled`, `interval_minutes`, `next_scan_at`,
`last_scan_at`, `last_status`, `last_error`, plus `locked_at`/`locked_by` for concurrency) and
**`bot_decisions`** (append-only, the full `BotDecision` object as `jsonb`, mirroring what's already
stored in the browser's `localStorage`-backed decision log today).

**Concurrency**: if a worker and the browser (or two workers) ever scanned for the same user at the
same time, both could read the same stale set of open trades, both independently pass Position
Manager/portfolio risk, and both open a trade â€” exceeding the intended risk limits. The fix is a
per-user advisory lock, not a database uniqueness constraint on open trades (which would incorrectly
break `ADD_TO_POSITION` â€” multiple same-side open positions are sometimes legitimate, see Mission 3).
`src/lib/scheduler/server-schedule-store.ts` implements this as a conditional `UPDATE`
(`claimScheduleLock`/`releaseScheduleLock`): a worker claims `bot_schedules.locked_at`/`locked_by`
before scanning, Postgres allows only one such `UPDATE` to succeed per row, and a lock older than
five minutes is treated as abandoned and reclaimable. Not called by anything yet â€” no worker exists.

**Known gap, documented rather than fixed this mission**: `reserveScanId()`
(`src/lib/bot/scan-id.ts`) is a purely local `localStorage` counter â€” unique per browser, not
globally unique across a browser and a future worker acting on the same user. A worker would need a
different scheme (a DB sequence or a UUID), not this mission's problem to solve.

**Service-role safety**: the service role key bypasses Row Level Security entirely â€” a worker using
it is not protected by the `auth.uid() = user_id` policies that protect the browser. All of the
safety, therefore, has to come from disciplined application code: always deriving `userId` from a
`bot_schedules` row (never trusting external input), always stamping `user_id` on every write
(`server-paper-trade-store.ts`/`server-bot-decision-store.ts` both take it as an explicit
parameter, never optional), and always going through `executeBotScan()` â€” never bypassing the risk
pipeline. See
[`../../docs/product/MISSION-6-SERVER-ARCHITECTURE-PREPARATION.md`](../../docs/product/MISSION-6-SERVER-ARCHITECTURE-PREPARATION.md)
for the full architecture review, schema, and what remains for Mission 7.

## VPS background worker (Mission 8)

The first real background execution service â€” a standalone process, `npm run worker`
(`src/worker/`), that wakes up, checks which users' `bot_schedules` are due, runs a scan for each,
persists the result, and sleeps, entirely independent of any open browser tab. It runs on the
architecture Mission 6 prepared and never executed, calling the exact same `executeBotScan()` +
`createServerExecutionContext()` pipeline the browser uses â€” no duplicated risk logic, no second
implementation of Strategy Engine/Position Manager/Portfolio Risk/Decision Intelligence.

**Lifecycle per due schedule**: claim the advisory lock (Mission 6) â†’ run one scan â†’ release the
lock with the outcome (`last_status`/`last_error`/`next_scan_at`). If the lock is already held by
another process, the worker logs why and skips that user for this cycle rather than waiting or
retrying. A scan that throws still releases its lock, with `status: "Error"` recorded, so a single
bad run can never leave a schedule stuck.

Run the web app and the worker side by side in two terminals â€” neither binds a port or depends on
the other:

```bash
npm run dev      # terminal 1
npm run worker   # terminal 2
```

Two things had to be fixed to make `npm run worker` actually runnable outside Next.js's own build
pipeline, both baked into the `worker` script itself: `NODE_OPTIONS=--conditions=react-server` (so
Mission 6's `import "server-only"` files don't throw â€” that safety mechanism depends on Next.js's
bundler specifically, not plain Node), and `tsx --env-file-if-exists=.env.local` (so local
development picks up the same config the web app uses, without requiring one to exist).

The worker uses the service role key (`SUPABASE_SERVICE_ROLE_KEY`) to read schedules and write scans
on a user's behalf â€” see [Server architecture
preparation](#server-architecture-preparation-mission-6) above for why that key is safe here (RLS
bypass, always-explicit `user_id`, never exposed to the browser). See
[`../../docs/product/MISSION-8-VPS-WORKER.md`](../../docs/product/MISSION-8-VPS-WORKER.md) for the
full lifecycle, deployment overview, environment variables, and verification â€” including a
lock-contention test proving a second concurrent claim for the same user is correctly rejected.

## Outcome analysis (Mission 11)

Every accepted `DecisionRecord` (Mission 7) carried `outcome: "Pending"` forever, even after its
linked trade closed. This mission classifies it: when a paper trade closes, its linked decision is
automatically judged **Win** (realised P/L `> ÂŁ0.01`), **Loss** (`< -ÂŁ0.01`), or **Neutral**
(everything in between, including exactly zero) â€” one shared threshold constant
(`NEUTRAL_PNL_THRESHOLD_GBP`), one pure function (`computeOutcomeUpdate()`,
`src/lib/decision-intelligence/outcome-analysis.ts`) that every trigger path below calls, so they
can never disagree.

**Two trigger paths, one shared function**: `DecisionHistoryProvider` sits inside
`PaperTradesProvider` in the provider tree, so it reacts to the trade list changing and classifies
the linked decision the moment a trade closes in the browser. The VPS worker (Mission 8) calls the
same logic once per poll cycle, for every user with a server schedule (Mission 10) â€” no new
permanent process, just one more step (`reconcileAllUsers()`) at the end of the existing cycle. A
purely browser-only user (no server schedule) is covered by their own browser's reconciliation
instead. Classification is idempotent by construction â€” a record's own `outcome !== "Pending"`
guard, not a separate flag, makes repeated reconciliation passes a safe no-op once a record is
classified.

**Rejected decisions are never classified** â€” they show `"N/A"` on the Decision Intelligence page,
never a `Pending` badge, so a reader never mistakes "no outcome applies" for "an outcome is coming."
The page gained an Outcome filter (All/Pending/Win/Loss/Neutral, which only ever matches accepted
decisions), three new columns (Realised P/L, Realised P/L %, holding duration), and a compact
**Outcome summary** panel â€” accepted/closed/pending counts, aggregate realised P/L, win/loss/neutral
counts, and a win rate computed as Wins Ă· (Wins + Losses) only, explicitly labelled as paper-trading
evidence, not a claim of strategy profitability.

New migration `0017_decision_outcomes.sql` adds five nullable columns to `decision_history`
(`realised_pnl`, `realised_pnl_percent`, `holding_duration_minutes`, `closed_at`,
`outcome_recorded_at`) plus a partial unique index enforcing one decision â†’ at most one trade at the
database level. **Confirmed not yet applied to the live Supabase project** â€” must be run manually
via the SQL Editor (same standing limitation as every migration since Mission 5: no direct
Postgres/SQL access in this environment). See
[`../../docs/product/MISSION-11-OUTCOME-ANALYSIS.md`](../../docs/product/MISSION-11-OUTCOME-ANALYSIS.md)
for the full outcome rules, reconciliation architecture, data-integrity guarantees, and verification
detail, including a real bug found and fixed via live testing (a `select=*` against a column the
migration hasn't added yet silently omits the key rather than erroring, which the original mapping
code didn't account for).

## Design intent

Calm, evidence-driven, professional. Dark theme with restrained accent colours (teal for
positive/active states, amber for caution/passive states, red for negative states, blue for
informational states). No crypto or gambling visual language, no profit claims. On Market
Intelligence specifically, colour is reserved for the two recommendation extremes (Strong Buy /
Strong Sell) and star ratings are monochrome â€” conviction is meant to read through layout and the
amount of supporting evidence, not colour intensity. The Intelligence Score follows the same rule:
plain monochrome bars, colour only on the two score-band extremes (Excellent / Avoid).

  was verified against an in-memory fake client, not real concurrent Postgres UPDATEs

  against a real, live browser session â€” no test account password was available this session; the

  separate systems, sharing no state â€” reconciling them into one was explicitly out of scope

  advances while the Dashboard tab is open; the worker (Mission 8) is the independent, tab-free path

  the resulting trade's scan id, portfolio risk metadata, and position metadata are persisted to
  Supabase, not the full candidate trace. Decision Intelligence (Mission 7) is the exception to
  this â€” its `decision_history` table records the full per-candidate detail, Supabase-scoped

  Rejected candidates are never classified, and no learning or strategy-optimisation logic reads
  these outcomes yet (that's Hermes, still out of scope)

  Supabase project â€” must be run manually via the SQL Editor before real outcome values persist
- Outcome analysis's interactive browser click-through (running a scan, closing a trade, watching
  the outcome badge/summary update live) was not verified this session â€” neither browser-automation
  tool available to this session was reachable (see `MISSION-11-OUTCOME-ANALYSIS.md`); pure-function
  and server-rendered-HTML checks were used instead

  `bot_decisions` table â€” they serve different purposes (live per-candidate history vs. a future
  worker's own scan log) and remain separate
- Portfolio risk limits are percentages of *starting* paper capital (a fixed number), not current
  portfolio value â€” a deliberate v1 simplification so the ceilings themselves don't drift
- Sector/category data is a small hardcoded lookup for this prototype's 5-instrument universe
  (`src/lib/mock/sectors.ts`), not a real classification standard
- No correlation-aware or cross-sector portfolio risk â€” "sector" is the only grouping considered
- Position Manager confidence/agreement comparisons only look at the latest *Bot*-sourced trade in
  an instrument + side â€” a position opened manually (Signal/Market Intelligence) has no comparable
  baseline, so the bot can't yet add to a manually-opened position with full confidence tracking
- AI, machine learning, or live/real market data feeding the Strategy Engine â€” three fixed
  deterministic rules, same result every time
- A fourth strategy (the current agreement-aggregation logic is written for exactly three)
- Strategy Engine metadata on Signal-sourced trades (a separate, older mock system, Build 0.1.0)
- Exit/close price provenance (only entry pricing records source/provider/timestamp so far)
- Per-user or per-account strategy configuration â€” the same three strategies and thresholds run
  for everyone
- Magic links, OAuth providers â€” email/password sign-up/sign-in (plus password reset) only
- Forcing re-authentication after a password reset (the recovery session is used as-is)
- Roles, teams, or shared/admin access â€” a paper trade belongs to exactly the user who created it
- Server-side session verification (middleware, protected API routes) â€” gating is client-side only

- Real broker connections, live order placement, or any real execution
- Real-time sync or multi-tab updates from Supabase (data loads once per page load, not polled)
- Automatic reconnection within a session after falling back to local storage or mock prices

- Technical indicators or model-generated scoring behind Market Intelligence or the Intelligence

- Persistence of Intelligence Scores at the time a trade was opened (scores are computed from

- Per-user scoping of the first-run import prompt (it's a single browser-wide flag)
- AI-generated signals or agents
- Financial advice of any kind
